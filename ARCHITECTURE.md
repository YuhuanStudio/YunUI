# YunUI architecture

How YunUI is structured as a *system* (not just a pile of components extracted
from one app). Read this before adding or reorganizing components.

## 1. Three layers, three entry points

YunUI is deliberately split so the core stays general-purpose and the app-domain
pieces are quarantined:

| Layer | Import | What lives here | Rule |
| --- | --- | --- | --- |
| **Core primitives** | `@yuhuanowo/yunui` | Button, Input, Card, Badge, Dialog, Select, Tabs, Switch, Checkbox, Slider, Tooltip, Popover, DropdownMenu, Modal, Sheet, layout (Flex/Grid/…), Table, Pagination, Breadcrumb… | Framework-agnostic, domain-neutral. Anything an app would want. |
| **Patterns** | `@yuhuanowo/yunui/patterns` | StatCard, PageHeader, Sidebar, Footer, FAQ, CodeBlock, BlogCard, page/media states… | Page-level compositions. Still generic, but bigger than a primitive. |
| **App domain** | `@yuhuanowo/yunui/ai` | ModelCard, ProviderIcon/ModelIcon, CapabilitySelector, ThinkingBlock, Navbar, LanguageSwitcher… | LLM/AI-gateway-shaped product UI. **Only import if you're building that kind of product.** |

Plus `@yuhuanowo/yunui/adapters` (framework injection) and
`@yuhuanowo/yunui/css` (the stylesheet).

> Origin note: YunUI was extracted *from* the Yunxin app. The `/ai` split is what
> keeps that origin from leaking into the core — a generic app stays on `.` +
> `/patterns`.

## 2. Adapter pattern (no framework lock-in)

Components carry **no copy** and bind **no framework**. Routing, images, i18n and
the icon base path are injected once at the root via `YunUIProvider`
(`@yuhuanowo/yunui/adapters`). Without a provider, zero-dependency defaults are
used (`<a>`/`<img>`/identity translation). So the same component renders in any
React app, and Yunxin's Next.js/next-intl wiring lives in Yunxin, not here.

## 3. Design tokens (the "system" in design system)

3 layers, generated by `scripts/gen-tokens.mjs` into `styles/tokens.css`:

1. **scheme** — raw palettes (`--scheme-{palette}-{100..1200}` + alpha), theme-independent.
2. **function** — role → palette map (`--function-{brand|accent|neutral|…}-*`),
   switchable at runtime via `data-brand` / `data-accent` / `data-neutral`.
3. **theme** — semantic tokens components consume
   (`--{role}-{background|on-background|border|alpha|solid|on-solid}-{strong|medium|weak}`),
   remapped per light/dark.

Runtime theming: `applyTheme()` / `useYunUITheme()` flip the `data-*` attributes.
**Opt-in brand mode** (`data-accent-source="brand"`) makes the whole interactive
layer follow the brand color; the default stays the monochrome Zinc look,
byte-for-byte. Legacy flat vars (`--bg-base`, `--text-primary`, …) still drive the
default look and are unchanged — the token system is additive.

## 4. Picking among overlapping components

The component JSDoc carries the canonical guidance (shows in IDE + props tables):

- **Dialogs:** `Dialog` (Radix) = accessible default · `Modal` = styled prop-driven
  (now traps focus) · `ConfirmModal*` = ready-made confirmations.
- **Selects:** `Select` (Radix, fixed JSX options) · `CustomSelect` (data-driven,
  optional search) · `Combobox` (type-ahead filter) · `SegmentedSelect` (2-5 inline).
- **Tabs:** `Tabs` (Radix, show/hide panels) vs `NavTabs` (controlled nav bar).

## 5. Conventions for new components

- Live in the right layer (core → `/patterns` → `/ai`); don't add app-domain
  things to the core.
- No hardcoded copy; text via `useYunUI().useT`. No `next/*` imports; use adapters.
- Consume semantic/`@theme` tokens so runtime theming reaches you; don't hardcode
  colors. Active/selected/focus states should follow the brand (see opt-in mode).
- Document or delete: every public export gets a docs page; the showcase demo's
  Code tab is auto-derived from the rendered children (no hand-written drift).

## 6. Known breaking-change backlog (needs a major / explicit sign-off)

Deferred on purpose — these break the public API, which the
[versioning policy](./CLAUDE.md#versioning-policy) (stay on `0.2.x`) forbids
without owner sign-off:

- Collapse overlapping components by **removing** the non-canonical ones (today
  they're kept and disambiguated by guidance, since each has real uses).
- Extract `/ai` into its **own published package** (today it's a sub-entry of the
  same package, which already isolates it without a breaking move).
- Rebuild `Modal` on Radix Dialog (today it's a self-contained portal that now
  traps focus, so the a11y reason is resolved).
